# Generation Config — fastedge-test

> **What this file is**: Structured instructions for generating `docs/` from source code.
> **How to use it**: Run `./fastedge-plugin-source/generate-docs.sh` which reads this file and invokes Claude per doc file.
> **Who reads this**: The generation script + Claude CLI. The plugin pipeline does NOT read this file.

---

## Global Rules

### Audience
These docs are for **consumers** of `@gcoredev/fastedge-test` — developers writing tests, building CI tooling, or integrating the debugger server into their workflow. They are NOT for contributors developing this repo.

This distinction drives what to include and exclude:
- **Include**: anything a consumer needs to use the package (public API, config options, exported types, consumer-facing endpoints)
- **Exclude**: internal implementation endpoints (e.g. Electron-only dialogs, VSCode extension integration points), internal type schemas used only by the server itself, contributor workflows, build system details
- **Test**: "Would someone installing this package from npm need to know this?" If no, leave it out.

Endpoints or features that exist solely for the VSCode extension or Electron shell to call are internal integration points, not consumer API. Do not document them.

### Style
- Technical prose only — no marketing language, no superlatives, no "easily" or "simply"
- Use TypeScript type signatures for all parameters and return types
- Use fenced code blocks for all examples (language-tagged)
- Use tables for structured data (endpoints, config fields, message types)
- Tables must use padded columns aligned to the widest cell in each column. Pad every cell with spaces so columns line up visually in raw markdown. The separator row dashes must match each column's padded width.
  Good: `| Variable             | Type     | Default |`
  Bad:  `| Variable | Type | Default |`
  Good: `| -------------------- | -------- | ------- |`
  Bad:  `|---|---|---|`
- Every code example must be self-contained and runnable
- Use `@gcoredev/fastedge-test` as the package name (not relative paths)

### Structure
- Every doc file starts with a level-1 heading and a one-line description
- Use level-2 headings for major sections, level-3 for subsections
- No table of contents — keep files navigable by heading structure alone
- End each file with a "See Also" section linking to related doc files

### Exclusions (apply to all files)
- No internal implementation details (how the code works internally)
- No source code file paths or line numbers
- No version history or changelog entries
- No references to `context/` or `CLAUDE.md` (those are internal developer docs)

### Accuracy
- Never hardcode version numbers, Node requirements, or other values that exist in source files (e.g. `package.json` `engines.node`). Instead, instruct the generator to read them from the source. Hardcoded values drift silently.
- All type signatures in docs must match the actual source code declarations

### Coverage Rule
- Every public export, endpoint, type, config option, and env var must appear in exactly one doc file
- If something is referenced in multiple files, one file is authoritative (has the full definition) and others link to it

**Accuracy notes for specific endpoints**:

**CRITICAL — Type accuracy rules (read types.ts carefully before generating)**:

The following types are commonly misrepresented. Copy them exactly from `server/runner/types.ts`:

1. **`FullFlowResult.hookResults`** is `Record<string, HookResult>` (an object keyed by hook name like `"onRequestHeaders"`), NOT `HookResult[]` (not an array). The server responds with `{ ok: true, ...fullFlowResult }` so the response is `{ ok: true, hookResults: Record<string, HookResult>, finalResponse: ..., calculatedProperties?: ... }`. This applies to BOTH `/api/execute` (Proxy-WASM) and `/api/send`.

2. **`FullFlowResult.finalResponse`** is an inline object `{ status: number; statusText: string; headers: Record<string, string>; body: string; contentType: string; isBase64?: boolean }`, NOT an opaque `HttpResponse` reference. Show the full inline type.

3. **`FullFlowResult.calculatedProperties`** is optional (`calculatedProperties?: Record<string, unknown>`), NOT required. The values come from `PropertyResolver.getCalculatedProperties()` and contain only derived `request.*` keys: `request.url`, `request.host`, `request.path`, `request.query`, `request.scheme`, `request.extension`, `request.method`. Do NOT use invented keys like `tenant_id` or `client_country` in `calculatedProperties` examples — use the real `request.*` keys.

4. **`HookResult.logs`** is `{ level: number; message: string }[]` (array of objects), NOT `string[]`.

5. **`HookResult.input` and `HookResult.output`** each have nested `request` and `response` sub-objects:
   ```
   input: {
     request: { headers: Record<string, string>; body: string };
     response: { headers: Record<string, string>; body: string };
     properties?: Record<string, unknown>;
   }
   ```
   They are NOT flat `{ headers, body }` objects. Same structure for `output`.

6. **`HookResult.returnCode`** is `number | null`, not just `number`.

7. **HTTP-WASM `execute()` response** (`POST /api/execute` when runner is HTTP-WASM): The server returns `{ ok: true, result }` where `result` is the full `HttpResponse` from `server/runner/IWasmRunner.ts`. This type has `contentType: string | null` (nullable, NOT just `string`) and includes a `logs: Array<{ level: number; message: string }>` field. Do NOT omit `logs` from the response type or example.

All JSON examples for `/api/call`, `/api/send`, and `/api/execute` (Proxy-WASM) MUST use these exact shapes. Do not simplify or flatten them.

- Response type schemas: `hook-result`, `hook-call`, `full-flow-result`, `http-request`, `http-response`
Group these into "Request Schemas" and "Response/Type Schemas" in the table.
